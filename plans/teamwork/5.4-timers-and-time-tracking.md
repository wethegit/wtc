# 5.4 Timers and Time Tracking ‚úÖ (in progress)

## Goal

Make time tracking fast from the TUI, especially for recurring project tasks and assigned tasks. Timers are managed **locally first** ‚Äî no API calls when starting/stopping. Timers are only submitted to Teamwork when the user explicitly sends them.

## Architecture Decision: Local-First

- **Local timer manager** (`src/teamwork/timers/local.ts`): start, stop, list, remove. Persisted to `wtc_cache_dir/teamwork-local-timers.json`.
- **Teamwork API layer** (`src/teamwork/timers.ts`): `startTimer()`, `pauseTimer()`, `getTimers()` ‚Äî called only during explicit submit.
- **No network calls** on start/stop. All state managed in cache file with `version: 1` schema.

## Start Timer From Task

- `ctrl+t` on a selected task toggles a local timer.
- If a timer is already running for that task, it stops. If another timer is running, it pauses the previous one and starts a new one.
- Status messages shown in the project header.
- Task metadata shows a `‚è±` indicator next to tasks with timers.

## Timer Tab

A dedicated Teamwork timers tab shows local timer entries:

- Running timer listed first (with flashing `‚è±` indicator)
- Stopped/pending timers below, sorted by start time (newest first)
- Read-only for now ‚Äî no submit/delete actions yet

## Visual Indicators

- Running timer: flashing `‚è±` (toggles `fg` between bright and dim every 800ms via `setInterval`)
- Stopped timer: dim `‚è±`
- No timer: nothing shown
