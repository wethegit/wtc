# 5.3 My Work Tasks

## Goal

Show tasks assigned to the current user so WTC can become a useful daily Teamwork dashboard.

## Scope

- Add a Teamwork read API for tasks assigned to the current user.
- Show assigned tasks in the Teamwork My Work tab.
- Start with all assigned tasks and defer advanced filters until real usage clarifies what matters.
- Support task selection with `↑` and `↓`.
- Support opening the selected task in the browser.

## API Shape

- Use a REST-style read function such as `getTeamworkMyTasks()`.
- The `get*` function should own its full read path.
- Do not add disk cache for My Work tasks initially because task lists change frequently.
- Keep shared HTTP/auth behavior in `src/teamwork/client.ts`.

## Display Fields

Start with the smallest useful task shape:

- ID
- name/title
- URL if available, or enough data to construct a URL
- status if available
- due date if available
- project/list context if available

## Future Actions

- Start timer for selected task.
- Open selected task in browser.
- Start branch workflow from selected task.
- Show branch and PR association when known.

## CLI Scope

```bash
wtc teamwork task mine
wtc teamwork task mine --json
wtc teamwork task open <taskId|taskUrl>
```

- `task mine` lists tasks assigned to the current user.
- `task open` opens a known task in the browser without requiring TUI selection.
- Keep task actions under `task` because they operate on task IDs or task URLs.

## Open Questions

- Which Teamwork endpoint best represents “tasks assigned to me”?
- Should completed tasks be hidden by default?
- Do we need pagination immediately, or can we start with Teamwork's default response size?
- Which task filters matter first: project, due date, status, priority, or all assigned tasks?
- Should the CLI command be `task mine`, `task assigned`, or `tasks mine`? Prefer singular `task` for consistency with `task open` unless that reads poorly in use.
